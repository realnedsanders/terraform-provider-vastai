<objective>
Create Terraform resources for team management: team (create/destroy), team roles (CRUD with permissions), and team members (invite/remove). These three resources form the RBAC system for Vast.ai teams.

Purpose: Team RBAC is an interconnected set of resources -- roles reference teams, members reference roles. Building them together ensures consistent patterns and proper handling of the API's asymmetric identifier usage (team role read/delete by name vs update by ID).

Output: 3 new resource directories under internal/services/ with resources, models, and unit tests.
</objective>

    - `TeamResource` struct with `client *client.VastAIClient`
    - Interface assertions: `resource.Resource`, `resource.ResourceWithImportState`
    - TypeName: `vastai_team`
    - Description: "Manages a Vast.ai team. Only one team can be managed per API key context. The destroy operation deletes the team associated with the current API key."
    - Schema attributes:
      - `id`: StringAttribute, Computed, UseStateForUnknown
      - `team_name`: StringAttribute, Required, ForceNew (team name appears immutable per research). Validator: stringvalidator.LengthAtLeast(1).
      - `timeouts`: timeouts block (create, read, delete)
    - Create: Call client.Teams.CreateTeam(teamName). Store returned ID.
    - Read: There is no single-team GET endpoint. Use the team ID stored in state. If team was destroyed externally, the list/context call will fail. Handle by checking API response -- if 404 or empty, remove from state.
      Implementation note: Since there's no direct read endpoint, Read can attempt to verify the team exists by calling ListMembers or ListRoles (which require team context). If these return a team-related error, remove from state. Alternatively, store the team data from Create and treat it as the source of truth with UseStateForUnknown on all computed fields.
    - Update: Not supported (ForceNew on team_name).
    - Delete: Call client.Teams.DestroyTeam() (parameterless).
    - ImportState: Import by ID string. Since there's no single-read endpoint, import sets the ID and Read populates what it can.

    **teamrole/models.go:**

    - `TeamRoleResource` struct with `client *client.VastAIClient`
    - Interface assertions: `resource.Resource`, `resource.ResourceWithImportState`
    - TypeName: `vastai_team_role`
    - Description: "Manages a Vast.ai team role with permission configuration."
    - Schema attributes:
      - `id`: StringAttribute, Computed, UseStateForUnknown. Description: "Numeric ID of the role (used for updates)."
      - `name`: StringAttribute, Required. Validator: stringvalidator.LengthAtLeast(1). Description: "Role name (used for read/delete operations)."
      - `permissions`: StringAttribute, Required. Description: "JSON string of permission configuration (per D-02). Example: {\"api\":{\"instance_read\":{...}}}". Add a custom JSON validity validator.
      - `timeouts`: timeouts block
    - Create: Parse permissions string to json.RawMessage. Call client.Teams.CreateRole(name, perms). Store returned ID and name.
    - Read: Call client.Teams.GetRole(name) (read uses NAME per Pitfall 3). Update ID, name, permissions in state.
    - Update: Call client.Teams.UpdateRole(id_as_int, name, perms) (update uses ID per Pitfall 3). The id field is the numeric identifier. Parse state.ID to int for the API call.
    - Delete: Call client.Teams.DeleteRole(name) (delete uses NAME per Pitfall 3).
    - ImportState: Import by numeric ID string. In Read, if only ID is available (no name), call ListRoles to find the role by ID and resolve the name. Then use name for subsequent GetRole call.

    Test files: Follow ssh_key_test.go pattern. For team: test create, destroy (verify parameterless DELETE). For team role: test the asymmetric ID usage -- create returns ID, read uses name, update uses ID, delete uses name.
